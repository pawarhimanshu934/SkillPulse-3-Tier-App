# Troubleshooting Guide: Gateway Connection Timeouts (50% Success/Failure)

This document explains the issue where accessing the application via the Gateway resulted in a connection timeout (`upstream connect error or disconnect/reset before headers. reset reason: connection timeout`) approximately 50% of the time, while working perfectly for the remaining 50%.

---

## 1. The Symptom
When accessing the application using the Gateway LoadBalancer URL:
```
http://aeb3fc5c84b4341cd9c42818a7de886e-945130905.us-east-1.elb.amazonaws.com/
```
* **50% of requests**: Loading perfectly fine.
* **50% of requests**: Failed with:
  > *upstream connect error or disconnect/reset before headers. reset reason: connection timeout*

---

## 2. The Root Cause

The issue is caused by a **Security Group restriction** on cross-node pod-to-pod TCP communication below port `1025` when using the AWS VPC CNI.

### EKS Pod & Node Distribution
The EKS cluster is deployed across two subnets in different Availability Zones (`us-east-1a` and `us-east-1b`). Consequently, pods are distributed across two different nodes (EC2 instances):

* **Node 1 (`ip-10-0-1-119.ec2.internal`)**:
  * `skillpulse-gateway-7967bcf9bc-l85pj` (IP: `10.0.1.167`)
  * `frontend-deployment-59d97748c9-5f5fs` (IP: `10.0.1.182`) — *Frontend replica 1*
* **Node 2 (`ip-10-0-2-181.ec2.internal`)**:
  * `frontend-deployment-59d97748c9-44qjg` (IP: `10.0.2.137`) — *Frontend replica 2*

### Why the 50% Failure Rate Occurs
1. **The Gateway API routes requests** round-robin/randomly between the two frontend pods.
2. **Success Case (50% of requests)**: 
   * When `skillpulse-gateway` on **Node 1** sends a request to Frontend replica 1 (`10.0.1.182:80`) which is also on **Node 1**, the communication is entirely local within the node's virtual interface. The traffic never leaves the host, completely bypassing external AWS Security Group rules.
3. **Failure Case (50% of requests)**: 
   * When `skillpulse-gateway` on **Node 1** sends a request to Frontend replica 2 (`10.0.2.137:80`) on **Node 2**, the traffic crosses nodes.
   * By default, the EKS shared node security group (`sg-0f0bfbe2133c373d1`) generated by the `terraform-aws-modules/eks` module only allows **node-to-node (inter-node) ingress on ephemeral ports `1025-65535`**.
   * Because the frontend pods run on **port `80`** (which is less than `1025`), the security group blocks the incoming request on Node 2.
   * This causes the Gateway to time out waiting for a connection, producing the `connection timeout` reset reason.

> [!NOTE]
> The backend pods (`backend-service`) run on port **`8080`**. Since `8080` falls within the allowed ephemeral range of `1025-65535`, cross-node communication to the backend service works without issue.

---

## 3. The Solution

The permanent solution is to add a self-referential ingress rule to the EKS node security group that opens all ports and protocols for inter-node communication. This is a standard requirement for Kubernetes clusters using the AWS VPC CNI where pods receive direct VPC IP addresses.

### Terraform Fix

Add `node_security_group_additional_rules` in your `eks.tf` module definition:

```hcl
module "eks" {
  source  = "terraform-aws-modules/eks/aws"
  version = "~> 21.0"

  # ... other EKS configurations ...

  # Allow all traffic between nodes in the cluster
  node_security_group_additional_rules = {
    ingress_self_all = {
      description = "Node to node all ports/protocols"
      protocol    = "-1"
      from_port   = 0
      to_port     = 0
      type        = "ingress"
      self        = true
    }
  }
}
```

### Applying the Changes
To apply the fix and resolve the timeouts immediately:
1. Navigate to the `terraform` directory:
   ```bash
   cd terraform
   ```
2. Run Terraform apply to update the security group rule:
   ```bash
   terraform apply
   ```
Once applied, the EKS node security group will allow Node 1 to communicate with Node 2 on all ports (including port 80), eliminating the 50% timeout failure rate.
